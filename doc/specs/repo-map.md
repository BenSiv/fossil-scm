# Repository Map

This document defines the preferred repository layout and what each top-level
area is responsible for.

## Top-Level Layout

- `src/`: application source code
- `dep/`: third-party and vendored dependencies
- `dev/`: developer tooling, scripts, editor settings
- `doc/`: project documentation
- `pub/`: distributable/public assets and packaging material
- `tst/`: tests and test helpers
- `scm/`: source-control metadata files tracked with the repo
- `cfg/`: shared configuration that does not clearly belong to `dev/` or `bld/`
- `bld/`: build scripts and build configuration inputs
- `out/`: generated build intermediates and temporary build output
- `bin/`: generated binaries/executables
- `log/`: generated build/configure logs

## File Placement Rules

- Keep source-control metadata in `scm/`.
  Current examples: `scm/manifest`, `scm/manifest.uuid`, `scm/manifest.tags`,
  `scm/VERSION`.
- Keep build recipes and templates in `bld/` (for example: makefile templates,
  legacy build makefiles, configure/build definitions).
  Current examples: `bld/auto.def`, `bld/Makefile.in`,
  `bld/Makefile.classic`, `bld/Makefile.osx-jaguar`.
- Keep generated files out of source directories.
- Treat `out/` and `bin/` as build output directories and ignore them from
  version control.
- Treat `log/` as generated output and ignore it from version control.
- Keep root directory minimal. Root should primarily contain project entrypoints
  and major project directories.

## Root Directory Policy

- Allowed in root: `configure`, `Makefile`, and core project folders (`src`,
  `dep`, `dev`, `doc`, `pub`, `tst`, `scm`, `cfg`, `bld`, `out`, `bin`,
  `log`).

- Avoid placing guides, metadata, or generated artifacts directly in root when
  a dedicated directory exists.

## Special Cases

- `configure`: keep in root as the main entrypoint, backed by build config in
  `bld/`.
- `out/autoconfig.h`: generated configure header; treat as build output.
- `log/config.log`: generated by configure; treat as transient log output.
